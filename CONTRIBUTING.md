# Contributing to Trussium

Thank you for your interest in contributing to Trussium.

Our goal is to build a cloud-native AI runtime that is reliable, extensible, and welcoming to contributors from across the AI and cloud-native communities.

Every contribution—whether documentation, bug fixes, new features, or architectural discussions—is appreciated.

---

# Code of Conduct

All contributors are expected to participate respectfully and professionally.

We are committed to fostering an open, inclusive, and collaborative community.

---

# Before You Start

Before contributing, please familiarize yourself with:

- Project Vision
- Roadmap
- Architecture
- Architecture Decision Records (ADRs)

Understanding these documents helps ensure contributions align with the long-term direction of the project.

---

# Ways to Contribute

Contributions include:

- Bug reports
- Documentation improvements
- Feature implementations
- Performance improvements
- Provider integrations
- Kubernetes improvements
- Testing
- Examples
- SDK improvements

Not every contribution needs to involve writing code.

---

# Reporting Issues

Before opening an issue:

- Search existing issues.
- Provide a clear description.
- Include reproduction steps.
- Include relevant logs where applicable.

---

# Proposing Features

Major features should begin with a discussion.

Architectural changes may require a new Architecture Decision Record (ADR).

The goal is to reach architectural alignment before implementation.

---

# Development Workflow

1. Fork the repository.
2. Create a feature branch.
3. Implement your changes.
4. Add or update tests.
5. Update documentation if necessary.
6. Ensure all checks pass.
7. Submit a Pull Request.

---

# Commit Messages

Trussium follows the Conventional Commits specification.

Examples:

```text
feat(runtime): add provider registry

fix(logging): correct JSON formatter

docs: update architecture guide

test(provider): improve coverage

refactor(config): simplify loader

ci: automate releases
```

---

# Pull Requests

A good pull request should:

- Have a clear description.
- Focus on a single logical change.
- Include tests when appropriate.
- Update documentation where necessary.

Large pull requests should be discussed before implementation.

---

# Coding Standards

We value:

- Readability
- Simplicity
- Consistency
- Maintainability
- Testability

Code should favor clear abstractions over clever implementations.

---

# Documentation

Documentation is considered a first-class contribution.

Significant architectural changes should update:

- Architecture
- ADRs
- README
- Relevant documentation

---

# Testing

All new functionality should include appropriate tests.

Where practical, bug fixes should include regression tests.

---

# Community

Questions, discussions, and constructive feedback are encouraged.

Healthy technical discussion leads to better software.

---

# Thank You

Thank you for helping build Trussium.

Every contribution helps move the project toward becoming a reliable, cloud-native runtime for AI applications.
